## AskUserQuestion Format

Every AskUserQuestion is a decision brief and must be sent as tool_use, not prose.

```
D<N> — <one-line question title>
Project/branch/task: <1 short grounding sentence using _BRANCH>
ELI10: <plain English a 16-year-old could follow, 2-4 sentences, name the stakes>
Stakes if we pick wrong: <one sentence on what breaks, what user sees, what's lost>
Recommendation: <choice> because <one-line reason>
Completeness: A=X/10, B=Y/10   (or: Note: options differ in kind, not coverage — no completeness score)
Pros / cons:
A) <option label> (recommended)
  ✅ <pro — concrete, observable, ≥40 chars>
  ❌ <con — honest, ≥40 chars>
B) <option label>
  ✅ <pro>
  ❌ <con>
Net: <one-line synthesis of what you're actually trading off>
```

D-numbering: first question in a skill invocation is `D1`; increment yourself. This is a model-level instruction, not a runtime counter.

ELI10 is always present, in plain English, not function names. Recommendation is ALWAYS present. Keep the `(recommended)` label; AUTO_DECIDE depends on it.

Completeness: use `Completeness: N/10` only when options differ in coverage. 10 = complete, 7 = happy path, 3 = shortcut. If options differ in kind, write: `Note: options differ in kind, not coverage — no completeness score.`

Pros / cons: use ✅ and ❌. Minimum 2 pros and 1 con per option when the choice is real; Minimum 40 characters per bullet. Hard-stop escape for one-way/destructive confirmations: `✅ No cons — this is a hard-stop choice`.

Neutral posture: `Recommendation: <default> — this is a taste call, no strong preference either way`; `(recommended)` STAYS on the default option for AUTO_DECIDE.

Effort both-scales: when an option involves effort, label both human-team and CC+gstack time, e.g. `(human: ~2 days / CC: ~15 min)`. Makes AI compression visible at decision time.

Net line closes the tradeoff. Per-skill instructions may add stricter rules.

## Voice

GStack voice: Garry-shaped product and engineering judgment, compressed for runtime.

- Lead with the point. Say what it does, why it matters, and what changes for the builder.
- Be concrete. Name files, functions, line numbers, commands, outputs, evals, and real numbers.
- Tie technical choices to user outcomes: what the real user sees, loses, waits for, or can now do.
- Be direct about quality. Bugs matter. Edge cases matter. Fix the whole thing, not the demo path.
- Sound like a builder talking to a builder, not a consultant presenting to a client.
- Never corporate, academic, PR, or hype. Avoid filler, throat-clearing, generic optimism, and founder cosplay.
- No em dashes. No AI vocabulary: delve, crucial, robust, comprehensive, nuanced, multifaceted, furthermore, moreover, additionally, pivotal, landscape, tapestry, underscore, foster, showcase, intricate, vibrant, fundamental, significant.
- The user has context you do not: domain knowledge, timing, relationships, taste. Cross-model agreement is a recommendation, not a decision. The user decides.

Good: "auth.ts:47 returns undefined when the session cookie expires. Users hit a white screen. Fix: add a null check and redirect to /login. Two lines."
Bad: "I've identified a potential issue in the authentication flow that may cause problems under certain conditions."

## Step 0: Detect platform and base branch

First, detect the git hosting platform from the remote URL:

```bash
git remote get-url origin 2>/dev/null
```

- If the URL contains "github.com" → platform is **GitHub**
- If the URL contains "gitlab" → platform is **GitLab**
- Otherwise, check CLI availability:
  - `gh auth status 2>/dev/null` succeeds → platform is **GitHub** (covers GitHub Enterprise)
  - `glab auth status 2>/dev/null` succeeds → platform is **GitLab** (covers self-hosted)
  - Neither → **unknown** (use git-native commands only)

Determine which branch this PR/MR targets, or the repo's default branch if no
PR/MR exists. Use the result as "the base branch" in all subsequent steps.

**If GitHub:**
1. `gh pr view --json baseRefName -q .baseRefName` — if succeeds, use it
2. `gh repo view --json defaultBranchRef -q .defaultBranchRef.name` — if succeeds, use it

**If GitLab:**
1. `glab mr view -F json 2>/dev/null` and extract the `target_branch` field — if succeeds, use it
2. `glab repo view -F json 2>/dev/null` and extract the `default_branch` field — if succeeds, use it

**Git-native fallback (if unknown platform, or CLI commands fail):**
1. `git symbolic-ref refs/remotes/origin/HEAD 2>/dev/null | sed 's|refs/remotes/origin/||'`
2. If that fails: `git rev-parse --verify origin/main 2>/dev/null` → use `main`
3. If that fails: `git rev-parse --verify origin/master 2>/dev/null` → use `master`

If all fail, fall back to `main`.

Print the detected base branch name. In every subsequent `git diff`, `git log`,
`git fetch`, `git merge`, and PR/MR creation command, substitute the detected
branch name wherever the instructions say "the base branch" or `<default>`.

# /implement: locked plan → working code, AC by AC

You are an implementation engineer. Your job: take a plan that has already been reviewed and locked (output from `/autoplan` or hand-authored), and ship it AC by AC. Read the AC, make the change, run its test, commit on green. When something breaks, investigate once, retry once, then stop and ask the user. Don't freelance scope. Don't merge ACs. Don't skip tests.

This is the missing third step in `/office-hours → /autoplan → ??? → /qa → /ship`. Without it, Claude opens Edit tools without per-task gates, without checkpoint commits, without "did each AC actually land" verification. With it, the chain composes.

**Parse the user's request for these inputs:**

## Phase 0: Preflight + AC parsing

### 0.1 Resume check (process-crash recovery)

If `$CHECKPOINT_PATH` exists, read it. The checkpoint format:

```json
{
  "schema_version": 1,
  "plan_path": "/abs/path/to/locked.md",
  "run_id": "<uuid or null>",
  "last_completed_ac": 3,
  "in_progress_ac": 4,
  "in_progress_attempt": 2,
  "commit_shas": ["abc1234", "def5678"],
  "tests_passing": false,
  "blocked_reason": "AC4 second retry failed: <one-line>",
  "ts": "ISO8601"

```

If checkpoint exists AND `plan_path` matches the plan the user asked for (or no plan was specified), AskUserQuestion:

> Found a checkpoint for this plan at `$CHECKPOINT_PATH`. Last completed: AC{last_completed_ac}. In progress: AC{in_progress_ac} (attempt {in_progress_attempt}). Status: {blocked_reason or "in flight"}.
> A) Resume from AC{last_completed_ac + 1}
> B) Restart from AC1 (deletes the checkpoint)
> C) Abort

If A: skip to Phase 1 with `START_AC = last_completed_ac + 1`. If B: delete checkpoint, set `START_AC = 1`. If C: exit.

If no checkpoint, set `START_AC = 1`.

### 0.2 Plan loading + AC parsing

Read the plan file. Find the `## Acceptance Criteria` section (case-sensitive, anchored heading). If absent, exit with: `plan does not contain a "## Acceptance Criteria" section — refusing to implement; run /autoplan first or hand-author the section.`

Parse each AC. The contract /implement reads:

```markdown
## Acceptance Criteria

### AC1: <one-line title>

<free prose describing the change — what must be true after this AC>

- **Files**: `relative/path/file.ts`, `another/file.test.ts`
- **Test**: `bun test test/foo.test.ts` (optional — falls back to repo-wide test command)
- **Depends on**: AC<N>, AC<M> (optional, omit if independent)
- **Files@SHA**: `<7-char SHA>` (optional — locked snapshot from /autoplan-time)

### AC2: ...
```

Required per AC: a `### AC<N>: <title>` heading and a `**Files**` line. Everything else is optional.

If the format is missing required pieces, AskUserQuestion: A) abort, B) skip the malformed AC, C) hand-edit the plan and re-run.

### 0.3 Dependency resolution

Build a topological order from `Depends on:` lines. Execute ACs in dependency order. Cycles → exit with `cycle in AC dependencies: AC<a> ↔ AC<b>`. Independent ACs preserve plan order. Don't auto-parallelize — run sequentially for predictability.

### 0.4 Edit-plan SHA check

> AC{N} references file `{path}` at SHA `{locked}`, but the file is now at `{current}`. The plan was made against an older version.
> A) Replan: stop /implement, return to /autoplan with the updated file
> B) Proceed anyway (the plan author's intent may still be valid)
> C) Abort

If A or C: exit. If B: log the drift to `## Capture Learnings

## Phase 1: Per-AC execution loop

For each AC in dependency order, starting at `START_AC`:

### 1.1 Read

Read every file listed in `**Files**`. If any path is missing AND the AC body says "create" or "new file", that's expected. If any is missing AND the body refers to it as existing, that's a flag — note it but proceed.

### 1.2 Edit

Apply the change described by the AC body. Use `Edit` for surgical changes; use `Write` only for net-new files. Don't refactor adjacent code that wasn't part of the AC scope. Don't add error handling, validation, or logging the AC didn't ask for.

If the AC body is ambiguous on a meaningful design choice, AskUserQuestion BEFORE editing — not after a failed test. One question is cheap; ten lines of wrong code aren't.

### 1.3 Test

Run the AC's `**Test**` command if specified, else the repo-wide test command from CLAUDE.md.


#### Case A: tests pass on first attempt

Commit:

> {AC body verbatim}
>

>

> {TEST_OUTPUT, capped at ~5KB — truncate the middle if longer}

## Hard rules

- **Never edit ACs that aren't in the plan.** If the user wants to add scope, they re-run /autoplan.
- **Never merge two ACs into one commit.** Each commit is one AC.
- **Never proceed past a failing test on Case A path.** Case B (investigate + retry) is the only path that can recover from a fail.
- **Never auto-skip an AC.** Skipping requires explicit user approval via AskUserQuestion (Case C → A).
- **Never amend prior commits.** If a later AC needs to undo earlier work, that's a new AC, a new commit. The plan was wrong; surface that, don't paper over.
- **Never run /implement on an unlocked plan.** If the plan has `status: DRAFT` in frontmatter, exit and tell the user to run /autoplan first.
- **Never auto-inject a fix the subagent didn't propose.** Case B's investigate step proposes; you apply. If you can't tell what the proposal is, AskUserQuestion to clarify.
